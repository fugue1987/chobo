# chobo 用量计量 CRM —— 交付与数据库说明

> 给接入方的一份完整说明:**怎么把 CRM 跑起来**,以及**库里每张表、每个字段是什么意思**(便于你们用自己的 admin 直接读这个库做展示)。
> 整个 CRM 不需要 Docker、不连任何外部地址,**所有数据都进你们自己的 Postgres**。

---

## 0. 运行环境(就两样)

| 要求 | 说明 |
|------|------|
| **操作系统** | Linux(Debian/Ubuntu/CentOS 等任意发行版均可;macOS 也行)。 |
| **Node.js ≥ 20** | 这是一个 Node 服务,目标机必须装有 Node 20 或更高。`node -v` 确认。装法见 https://nodejs.org 或用 `nvm`。 |
| **PostgreSQL** | 你们自己的一台 Postgres(版本 12+ 均可),以及一个**空库**(下面第 2 步建)。CRM 只往这个库写,绝不碰你们其它库。 |

> 交付包里**已经带好了全部运行依赖**(`server/node_modules`,纯 JavaScript、跨平台),**不需要联网 `npm install`**。

---

## 1. 把 CRM 跑起来(手把手)

### 第 1 步:解压

```bash
tar xzf chobo-crm-bare-<版本>.tar.gz
cd chobo-crm
```

解压后目录长这样:

```
chobo-crm/
├── start.sh                 ← 启动脚本(自动算好内部路径,移到哪都能跑)
├── chobo-crm.env.example    ← 配置样例 → 复制成 chobo-crm.env 改两项
├── README.md                ← 简版说明
├── 交付指南.md              ← 本文件
├── price-seed.json          ← 价目表种子(决定每个模型怎么算钱)
├── server/                  ← CRM 程序本体(已编译,不是源码)
│   ├── dist/                ←   实际跑的就是 server/dist/server.js
│   ├── migrations/          ←   启动时自动建表/升级,无需手动建表
│   └── node_modules/        ←   运行依赖(已含,无需联网安装)
├── contracts/               ← 数据契约(程序内部校验用)
└── web/                     ← 自带看板的静态页面(可选;不用看板可忽略)
```

### 第 2 步:准备一个空库

用你们的 Postgres 建一个空库和一个账号(不用建表,CRM 启动会自动建):

```sql
CREATE DATABASE chobo;
CREATE USER chobo WITH PASSWORD '改成你们的强密码';
GRANT ALL PRIVILEGES ON DATABASE chobo TO chobo;
```

> 库名/账号随你们定,记住连接信息下一步要填。

### 第 3 步:填配置

```bash
cp chobo-crm.env.example chobo-crm.env
```

编辑 `chobo-crm.env`,**只需填两项**(原文粘贴,不要加引号):

```bash
# 指向第 2 步那个库
CHOBO_DATABASE_URL=postgresql://chobo:你们的密码@127.0.0.1:5432/chobo

# ingest 闸门:自定一串强随机字符。业务服务上报时要带同一串,否则被拒。
CHOBO_INGEST_SECRET=一串足够长的强随机字符
```

其余(监听地址端口等)留默认即可,默认监听 `0.0.0.0:8787`。

### 第 4 步:启动

```bash
./start.sh
```

看到这样一行日志就表示**起来了**(已自动建表、灌好价目表):

```
... "priceVersion":"2026-06-26a", "seedInserted":true, ... "msg":"chobo CRM up"
```

- 自带看板:浏览器打开 `http://<本机IP>:8787/`。
- 不想用看板?**直接连第 2 步那个库读数据即可**(字段含义见第 2 部分)。

> 报错不会被吞掉,日志里会写清原因(比如库连不上、Node 版本过低)。卡住就把启动日志发回来。

### 第 5 步:让业务服务把数据打过来

在你们已经接了 chobo SDK 的业务服务(如 node-ai-proxy / python-lesson-parser)的 `.env` 里加三行:

```bash
CHOBO_INGEST_URL=http://<CRM所在主机IP>:8787/v1/events
CHOBO_INGEST_SECRET=<与 chobo-crm.env 里【完全相同】>
CHOBO_SPOOL_DIR=./.chobo-spool
```

重启业务服务。之后**每次大模型调用都会异步上报到这台 CRM**(不阻塞、不拖慢业务;CRM 挂了也只丢计量、不影响业务)。

### 让它常驻(生产)

`start.sh` 是前台进程,生产建议用 systemd 守护。新建 `/etc/systemd/system/chobo-crm.service`:

```ini
[Unit]
Description=chobo CRM
After=network.target

[Service]
WorkingDirectory=/opt/chobo-crm
ExecStart=/opt/chobo-crm/start.sh
Restart=always
User=chobo

[Install]
WantedBy=multi-user.target
```

```bash
sudo systemctl enable --now chobo-crm
sudo journalctl -u chobo-crm -f       # 看日志
```

> 安全提醒:看板没有登录页。若 8787 会暴露到公网,请前置 nginx 加 basic-auth(仅放行 `POST /v1/events`),或用防火墙只放通业务服务来源 IP。

## 2. 数据库表与字段说明(给你们的 admin 用)

库里有 4 张业务表。**你们要展示的几乎都在 `usage_events` 这一张表里** —— 一行 = 一次大模型调用,用量和算好的成本都在这。

### 2.1 主表 `usage_events`

> 一行就是一次调用的完整账目。下面按用途分组列出所有字段。金额列都是 `numeric(18,8)`(8 位小数,**别用 float 读,会丢精度**)。

**① 归因(谁调用的 —— 这正是网关看不到、本产品补上的"每个终端用户"维度)**

| 字段 | 类型 | 含义 |
|------|------|------|
| `account` | text | 哪个接入方/应用(多租户维度)。例如 `adopter-a`。可空。 |
| `user_id` | text | **终端用户 id**(谁用的)。可空。 |
| `org_id` | text | 用户所属组织/机构。可空。 |
| `project` | text | 项目维度。可空。 |
| `identity_source` | text | 身份从哪来:`header` / `jwt` / `default`(粗粒度默认值) / `missing`(没识别到身份,异常信号)。 |

**② 时间与服务**

| 字段 | 类型 | 含义 |
|------|------|------|
| `start_time` | timestamptz | 调用**发生**时刻。**出账、做时间趋势用这个**。 |
| `end_time` | timestamptz | 调用结束时刻。可空。 |
| `latency_ms` | integer | 调用耗时(毫秒)。可空。 |
| `service` | text | 哪个被插桩的服务发起的(如 `node-ai-proxy`)。 |
| `created_at` | timestamptz | **入库**时刻(≠ 调用时刻)。 |

**③ 模型与操作**

| 字段 | 类型 | 含义 |
|------|------|------|
| `provider` | text | **计价通道**(不是模型厂商):如 `doubao`(直连)、`newapi`(网关)。表示"按谁的价算"。 |
| `operation` | text | 操作类型:`chat` / `image` / `video` / `embedding`。 |
| `request_model` | text | 请求时用的模型名。 |
| `response_model` | text | 响应实际返回的模型名。可空。 |

**④ 用量(token / 张数)**

| 字段 | 类型 | 含义 |
|------|------|------|
| `input_tokens` | integer | 输入 token。 |
| `output_tokens` | integer | 输出 token。 |
| `total_tokens` | integer | 总 token。 |
| `cached_tokens` | integer | 命中缓存的输入 token(按缓存价,更便宜)。 |
| `reasoning_tokens` | integer | 推理/思维链 token。 |
| `image_count` | integer | 生成图片张数(`image` 操作)。 |
| `input_text_tokens` | integer | 输入里的**文本** token(逐模态拆分,如 gpt-image-2)。 |
| `input_image_tokens` | integer | 输入里的**图像** token(逐模态拆分)。 |
| `usage_source` | text | 用量从哪来:`measured`(真实读到)/ `estimated`(估算,**已标记,别当精确值**)/ `none`。 |

**⑤ 成本(已算好,核心)**

| 字段 | 类型 | 含义 |
|------|------|------|
| `total_cost` | numeric(18,8) | **该次调用的总成本**(各项之和)。**`NULL` 表示当时没有该模型的价 → 诚实未计价,别当成 0。** |
| `input_cost` | numeric(18,8) | 输入部分成本。 |
| `output_cost` | numeric(18,8) | 输出部分成本。 |
| `cache_cost` | numeric(18,8) | 缓存读取部分成本。 |
| `cost_breakdown` | jsonb | 成本逐项明细(按模态/费率项拆),展示"为什么是这个价"用。 |
| `currency` | text | 币种,如 `CNY` / `USD`。**⚠ 永不跨币种相加** —— 统计务必 `GROUP BY currency` 或分开展示。 |
| `price_table_version` | text | 这次用的哪一版价目表(可审计)。 |

**⑥ 状态与审计**

| 字段 | 类型 | 含义 |
|------|------|------|
| `event_id` | text(主键) | 事件唯一 id。**幂等去重键** —— 重投同一 id 不会重复计费。 |
| `status` | text | 调用结果:`success` / `failure`。 |
| `error_type` | text | 失败类型。可空。 |
| `finish_reason` | text | 结束原因。可空。 |
| `request_id` | text | 上游请求 id(可空,用于对账)。 |
| `parent_id` | text | 父事件 id(多阶段调用串联,可空)。 |
| `sdk_lang` | text | 上报 SDK 语言:`python` / `node`。 |
| `sdk_version` | text | SDK 版本。 |

> **预留列(v1 不填,可忽略):** `newapi_cost`、`cost_delta`、`recon_status` —— 给"与网关对账"功能预留,当前恒为空。

**哪些列一定有 / 可能为空:**
- **一定有**:`event_id`、`identity_source`、`start_time`、`service`、`provider`、`operation`、`request_model`、`usage_source`、`status`、`currency`、`created_at`。
- **看情况**:`user_id`/`account`(取决于业务有没有注入身份)、各 token(`chat` 才有)、`total_cost`(有价才有)、图像几列(`image` 才有)。

### 2.2 其它三张表(一般不用动)

| 表 | 作用 |
|----|------|
| `event_payloads` | 可选留存的原始请求/响应载荷(`request_payload` / `response_payload`,jsonb),按 `event_id` 一对一关联 `usage_events`。默认只存元数据,不存完整正文。 |
| `price_table` | 价目表本身:每个 `(版本, provider, model, operation)` 的各项费率(`input_per_mtok` 等,单位"每百万 token")。想看"某模型怎么算钱"查这里。 |
| `model_aliases` | 模型名归一映射(把带版本号的实际模型名映射到价目里的规范名)。 |

> 还有一张 `schema_migrations` 是程序内部记录迁移用的,**请勿改动**。

### 2.3 给 admin 的示例查询

**A. 按终端用户出账(分币种,绝不跨币种相加):**
```sql
SELECT account, user_id, currency,
       count(*)        AS calls,
       sum(total_cost) AS cost
FROM usage_events
WHERE total_cost IS NOT NULL
GROUP BY account, user_id, currency
ORDER BY cost DESC;
```

**B. 按模型看用量与成本:**
```sql
SELECT provider, request_model, currency,
       count(*)            AS calls,
       sum(total_tokens)   AS tokens,
       sum(total_cost)     AS cost
FROM usage_events
GROUP BY provider, request_model, currency
ORDER BY cost DESC NULLS LAST;
```

**C. 按天趋势(用 start_time):**
```sql
SELECT date_trunc('day', start_time) AS day, currency,
       count(*) AS calls, sum(total_cost) AS cost
FROM usage_events
GROUP BY day, currency
ORDER BY day;
```

**D. 某个终端用户的调用明细:**
```sql
SELECT start_time, provider, request_model,
       input_tokens, output_tokens, total_cost, currency, status
FROM usage_events
WHERE user_id = '某用户'
ORDER BY start_time DESC
LIMIT 200;
```

### 2.4 做展示时务必守的几条(计费正确性)

1. **永不跨币种相加。** 不同 `currency` 的 `total_cost` 不能直接 `sum` 到一起,要么 `GROUP BY currency`,要么按币种分区展示。
2. **`total_cost IS NULL` ≠ 0。** NULL 表示"当时没有这个模型的价、诚实未计价",不要显示成 ¥0。后续我们补了价可以重算回填。
3. **金额用 `numeric` 精度读**,别用浮点截断(列本身是 8 位小数)。
4. **`usage_source='estimated'`** 的行用量是估算的,展示时可加标识。
5. **出账时间用 `start_time`**(调用真实发生时刻),`created_at` 只是入库时刻。
6. 想只统计成功调用就加 `WHERE status='success'`(失败调用也有记录,看你们需求)。

## 以后新增模型价格(自助,无需联系我们)

模型频繁出新。给一个新模型计价,**不需要重启 CRM、不需要我们介入**,三步:

1. 编辑 `price-seed.json`:把顶层 `version` **改成一个更大的新版本号**(如 `2026-07-01a`),并在 `rows` 里**追加**这个新模型的价目行(保留原有全部行 —— 新版本是"整版快照")。
2. 跑 `./update-prices.sh`(写库 + 回填,幂等安全)。
3. 完。运行中的 CRM 会在 ≤ `CHOBO_PRICE_REFRESH_SEC` 秒内自动生效(默认 60s;若设为 0 则需重启进程)。

### 价目行字段(单位:每百万 token 的价 = per-mtok)

| 字段 | 含义 |
|------|------|
| `provider` | 计价渠道(**不是厂商**):经网关中转填网关名(如 `newapi`),直连填直连标识 |
| `model` | 规范模型名(与事件里 `request_model` 对齐;带版本后缀的用下方 `aliases` 归一) |
| `operation` | `chat` / `image` / `video` / `embedding` |
| `input_tier_max` | 输入分档上界;`0` = 不分档/兜底 |
| `input_per_mtok` / `output_per_mtok` | 输入 / 输出单价 |
| `cache_read_per_mtok` | 命中缓存的输入单价(可省) |
| `reasoning_per_mtok` | 单列的推理 token 单价(仅当上游把 reasoning 单独计费时填) |
| `text_input_per_mtok` | 仅图像 token 计价用(文本输入单价);此时 `input_per_mtok` 表图像输入单价 |
| `per_image` | 旧"按张"平价(有 token 计价就不用) |
| `currency` | 币种,如 `CNY` / `USD`。**看板永不跨币种相加**;按上游真实计价单位填 |

### 怎么找准 `(provider, model, operation)` 三元组(最易错)

定价是**精确匹配**这三元组。别猜 —— 直接查你自己的库,看事件真实带的是什么:

```sql
SELECT DISTINCT provider, request_model, operation, currency
FROM usage_events
WHERE request_model LIKE '%你的新模型关键词%';
```

按查出来的 `provider` / `request_model` / `operation` 去配价。若 `request_model` 带版本后缀(如 `xxx-260215`),在 `price-seed.json` 的 `aliases` 里加一条归一。

> 忘记加 alias 的后果:价目行永不命中 → 该模型成本继续记为 NULL(诚实留空,不是 0,也不会报错)。

### 完整实例:加 `doubao-seed-2.1-pro`(照这个模式加你自己的新模型)

这份包的 `price-seed.json` 里**已经带了一行真实样例** —— `doubao-seed-2.1-pro`。它就是照下面 4 步加进去的,你以后碰到新模型照抄这个套路即可:

**① 去该模型的官方价页抄价。** 豆包 Ark 控制台「模型定价」显示:推理输入 6、推理输出 30、缓存命中 1.2(元/百万 token,CNY)。
> 按小时的「缓存存储 0.017 元/百万token/时」是按**时间**计的存储费,本 CRM 按 token 计、不建模,忽略。

**② 从你自己的库查真实三元组。** 服务实际发的模型名是 `doubao-seed-2-1-pro-260628`:
```sql
SELECT DISTINCT provider, request_model, operation, currency
FROM usage_events WHERE request_model LIKE '%2-1-pro%';
-- → provider=doubao, request_model=doubao-seed-2-1-pro-260628, operation=chat
```

**③ 改 `price-seed.json`:** bump 顶层 `version`,`rows` 里加一行价目、`aliases` 里加一条归一(把带日期后缀的实际名映射到规范名):
```jsonc
// version: "2026-06-26a" → "2026-07-01a"
// rows 里加(单一价 → input_tier_max: 0):
{ "provider": "doubao", "model": "doubao-seed-2.1-pro", "operation": "chat",
  "input_tier_max": 0, "input_per_mtok": 6.0, "output_per_mtok": 30.0,
  "cache_read_per_mtok": 1.2, "currency": "CNY" }
// aliases 里加:
{ "provider": "doubao", "alias": "doubao-seed-2-1-pro-260628", "canonical": "doubao-seed-2.1-pro" }
```
> 若该模型有**输入长度分档**(不同输入 token 段不同价),就按档各加一行:相同 `provider/model/operation`、不同 `input_tier_max` + 各自价。

**④ 跑一条命令:**
```bash
./update-prices.sh
```
写库 + 回填历史 NULL,运行中的 CRM 在 ≤ `CHOBO_PRICE_REFRESH_SEC` 秒内自动生效(默认 60s)。之后该模型的调用就按 ¥6/¥30/¥1.2 逐终端用户计价了。

**轮到你加新模型时**(比如某个新出的模型),同样 4 步:去**那个模型**的官方价页抄价 → 从**你自己的库** `SELECT DISTINCT` 查它真实的三元组 → 加行(必要时加 alias)→ bump `version` → `./update-prices.sh`。**价格和 `provider`/`model`/`operation` 一律以你的实际为准**,不用问我们。

> 提醒:`version` 不 bump 就是 no-op(`update-prices.sh` 会提示"已在库,无改动")。`price-seed.json` 里现有的几行价目是**示例/起点**(含我们加的 `doubao-seed-2.1-pro` 样例),请按你们实际在用的模型与 `provider` 核对增删。

## 附:常见问题

| 现象 | 多半原因 |
|------|----------|
| 启动报 `CHOBO_DATABASE_URL is required` | `chobo-crm.env` 没改名,或没和 `start.sh` 在同一目录,或没填库地址 |
| `password authentication failed` | 库账号/密码不对,或第 2 步的用户没建/没授权 |
| `找不到 node` / 起来就退 | Node 版本 < 20,或 PATH 里没有 node |
| 看板能开但没数据 | 业务服务没配 `CHOBO_INGEST_URL`,或两边 `CHOBO_INGEST_SECRET` 不一致,或还没产生调用 |
| 端口被占 | 在 `chobo-crm.env` 里改 `CHOBO_PORT` |

日志不会静默吞错,卡住把启动日志发回来即可定位。
